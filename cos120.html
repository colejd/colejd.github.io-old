COS120 Help and Tips

This page is intended as a resource to help you avoid common point deductions as a COS120 student.

While I have your attention: pick up your homework! I made this page because every semester many students do not do this, and they miss my notes. Help me help you!

General Assignment Notes

  • All assignments must start with a comment stating your name.

  • You must submit your assignment with screenshots of the console showing what is asked for in the assignment. You will not receive credit if this is not done.

  • If you submit a screenshot of your code instead of printing it out, you will not receive credit. Code files are more or less just .txt files with a different extension, and Visual Studio has a print option just like Word. Similarly, please don't paste your code into Word and print it from there.

  • When using a StreamReader sr, you must close it with sr.Close() when you are done with it (at the end of the program).

Give your variables descriptive names

Suppose in the following code we have to choose a name from one of these lines to represent the average weight of a group of packages. Which one would you pick?

Dim n as Double
Dim apw as Double
Dim avgPkgWt as Double
Dim averagePackageWeight as Double

We want line 4, or averagePackageWeight, as the name. n is a bad name for any variable unless you're doing some mathematical formula or using it as a counting variable (in a loop, for example). apw is getting there, but it's still impossible to tell what it is from the name alone. avgPkgWt is better still, and it would be fine accompanied by a good comment. However, averagePackageWeight is the most desirable because it's self-documenting; that is, you can tell what the variable does without having to look at the comment next to its declaration. When in doubt, go with plain English. If the variable name ends up being very long, it's fine to abbreviate.

Comment your variable declarations

It's boring, but you need to do it to keep yourself organized. The less you have to think while programming, the better; the benefit of commenting your code well is that you don't have to worry about forgetting what certain things do. Comment as you go -- it will help you as you do the work and you won't have to take the time to go back through and do it when you're done.

You must comment all of your variable declarations on homework assignments. See the following:

Dim mediumPackageCount as Integer           'The total number of medium-sized packages
Dim sr as IO.StreamReader                   'Stream reader for file

Note that the comments tell you what the variables are for. For example let's see some comments we might write for a variable representing the sum of the weights of a set of packages:

Dim sumOfWeights as Double          'Sum                           'Bad comment
Dim sumOfWeights as Double          'Number                        'Really bad comment
Dim sumOfWeights as Double          'Sum of all package weights    'Good comment

Don't make variables class-level unless they have to be

Consider the following code example. We have two buttons, btnCompute and btnPrint. btnCompute computes some data, and btnPrint does something with that computed data.

Public Class Form1
    Dim classLevelNumber As Integer       'Class-level variable
    Dim y As Integer                      'Unnecessary class-level variable

    Private Sub buttonCompute_Click(ByVal sender As System.Object, ByVal e As System.EventArgs) Handles btnCompute.Click
        classLevelNumber = 5
    End Sub

    Private Sub buttonPrint_Click(ByVal sender As System.Object, ByVal e As System.EventArgs) Handles btnPrint.Click
        Dim x As Integer
        x = classLevelNumber
        y = classLevelNumber
    End Sub

End Class

classLevelNumber is an example of a variable that should be class-level. It is used in both buttonCompute_Click and buttonPrint_Click, so it cannot be declared in either of those functions. The only other place you can declare it is at the class level.

y is an example of a number that should not be class-level. It's only assigned to inside of buttonPrint_Click, so the declaration of y should be in that function, similar to how x is declared in the same spot.

Pay attention to formatting

When you create a new level of scope, you must indent. To review scope, see Microsoft's page on the subject. Consider the following, where each level of indentation (tab) is shown with a ⟹:

Private Sub SomeFunction()
⟹Dim x as Integer = 0
⟹Dim y as Integer = 0
⟹Dim z as Integer = 0
⟹
⟹If x = 0 Then
⟹⟹y = 0
⟹⟹If z = 0 Then
⟹⟹⟹'Some code here
⟹⟹End If
⟹Else
⟹⟹'Some other code here
⟹End If
End Sub

Each If/ElseIf/Else block creates a new level of scope. After the first If, all lines are indented once until we reach Else. Same from Else to EndIf. This lets you visually see that lines 7-9 will only execute if the If x = 0 statement is true, and that line 12 will only execute if that statement evaluates to false. Points will be taken for excessively messy submissions. Visual Studio automatically indents for you, so take advantage of it!

Cast strings as numeric types when using them as numbers

Consider the following code:

Dim someText as String = "5"
Dim x as Integer

x = someText            'Incorrect (someText is implicitly cast as integer)
y = CInt(someText)      'Correct (someText is explicitly cast as integer)

x (an integer) is set to the value of someText, which is "5" (a string). The problem is that the types aren't the same. Here, Visual Basic will do something called type coercion; it will try to get an integer from someText, which it will then assign as the value of x. It's nice that it does this for you, but in this course, type casting must be done explicitly with CInt, CDbl, etc. This is because doing it this way makes it much more obvious what the code is doing at first glance; additionally, if you're getting the string from an input field, it ensures that the value you're getting from what the user entered is of the type that you expected.

You can read more about implicit and explicit conversions on Microsoft's page on the topic.

Tips

  • Don't be afraid to Google functions if you don't know what they do. Coding languages as a whole are very, very well documented online.

  • Read the assignment carefully. If you're asked to implement functions given on the page or described in class and you don't do it, it typically results in you needing to resubmit with changes to get credit for that portion of the assignment.

  • Comment your code as you go. If you write some code and think "that was complicated", that's a good indicator that you should leave a comment for your future self.

  • If you absolutely have to email your code somewhere to print it out, just attach your source files to the email, and then open and print them on the other computer with Notepad. Don't copy the code and paste it into the email. It mangles the formatting and makes it extremely hard to read the code.

  • When using a StreamReader, the path you give to your text file is relative to your executable. So if you're in the debug configuration and your path is "someFile.txt", it's going to look for a file with that name in the debug directory. This is true of most (if not all) functions that take a path string in Visual Basic.
